Here's how TSDB compaction works end-to-end:

## 1. The `Compactor` Interface (`compact.go:52-77`)

```go
type Compactor interface {
    Plan(dir string) ([]string, error)
    Write(dest string, b BlockReader, mint, maxt int64, base *BlockMeta) ([]ulid.ULID, error)
    Compact(dest string, dirs []string, open []*Block) ([]ulid.ULID, error)
}
```

Three responsibilities:
- **Plan** — scan on-disk block directories and return groups that should be compacted together
- **Write** — persist a `BlockReader` (e.g. the in-memory head) into a new on-disk block
- **Compact** — merge multiple existing on-disk blocks into one new block

## 2. `LeveledCompactor` (`compact.go:79-93`)

The production implementation. Key fields:

```go
type LeveledCompactor struct {
    ranges                      []int64          // time ranges per compaction level
    chunkPool                   chunkenc.Pool
    mergeFunc                   storage.VerticalChunkSeriesMergeFunc
    enableOverlappingCompaction bool
    blockExcludeFunc            BlockExcludeFilterFunc
    // ...
}
```

**Planning** (`compact.go:279-328`): `plan()` sorts blocks by MinTime, then:
1. First picks overlapping blocks (vertical compaction)
2. If none, uses `selectDirs` for level-based compaction (excludes the newest block to allow backups)
3. As a fallback, compacts blocks with >5% tombstones

**Compact** (`compact.go:483-487`): Delegates to `CompactWithBlockPopulator`, which opens each block, merges their metas via `CompactBlockMetas`, then calls `write()`.

**write** (`compact.go:657+`): Creates a temp directory, writes chunks and index via `BlockPopulator.PopulateBlock`, then renames the temp dir to the final ULID-based path. On error, the temp dir is cleaned up.

## 3. How `DB` Triggers Compaction

### DB struct (`db.go:281-342`)

Key compaction-related fields:
```go
type DB struct {
    compactor      Compactor
    compactc       chan struct{}    // signal channel
    autoCompact    bool
    compactCancel  context.CancelFunc
    cmtx           sync.Mutex      // serializes compactions and deletions
    // ...
}
```

### Initialization (`db.go:990+`)

During `open()`, the DB creates a `LeveledCompactor` (or uses `opts.NewCompactorFunc` if provided):

```go
if opts.NewCompactorFunc != nil {
    db.compactor, err = opts.NewCompactorFunc(ctx, r, l, rngs, db.chunkPool, opts)
} else {
    db.compactor, err = NewLeveledCompactorWithOptions(ctx, r, l, rngs, db.chunkPool, ...)
}
```

### Three trigger paths

**A. Appender commit** (`db.go:1360-1372`): After every `Commit()`, if `head.compactable()` is true, a signal is sent on `compactc`:
```go
if a.db.head.compactable() {
    select {
    case a.db.compactc <- struct{}{}:
    default:
    }
}
```

**B. Periodic tick** (`db.go:1175-1243`): `DB.run()` is a background goroutine with two select cases:
- Every `BlockReloadInterval` — reloads blocks, then sends on `compactc`; also checks for stale series compaction
- On `<-db.compactc` — if `autoCompact` is true, calls `db.Compact(ctx)` with exponential backoff on failure

**C. Stale series** (`db.go:1204+`): If the ratio of stale series exceeds the threshold and the next regular compaction isn't imminent (within 10 min), triggers `CompactStaleHead()` immediately.

### `DB.Compact()` — the main flow (`db.go:1410-1506`)

Three phases, in order:

1. **Head compaction** — loops while `head.compactable()`:
   - Optionally delays start (`CompactionDelay`)
   - Creates a `RangeHead` for the compactable time range
   - Waits for overlapping appenders to finish
   - Calls `compactHead()` → `compactor.Write()` to persist head to a block
   - Truncates WAL up to the compacted max time

2. **OOO head compaction** — if head was compacted and out-of-order was ever enabled:
   - `compactOOOHead()` → `compactOOO()` creates one block per block-range interval from the OOO head
   - Each block is written via `compactor.Write()`, then blocks are reloaded and OOO WAL/chunks are truncated

3. **Block compaction** (`db.go:1719-1763`) — `compactBlocks()` loops:
   - Calls `compactor.Plan(db.dir)` to find compactable block groups
   - Calls `compactor.Compact(db.dir, plan, db.blocks)` to merge them
   - Reloads blocks after each compaction (which also applies retention)
   - Breaks early if the head becomes compactable again (head has priority)

### Control

- `DB.EnableCompactions()` / `DB.DisableCompactions()` toggle `autoCompact` under `autoCompactMtx`
- `compactCancel` cancels the compactor's context on DB shutdown, aborting in-progress compactions
