## How Prometheus Metrics Scraping & Collection Works

Based on the fixture code in `testdata/fixtures/go/`, here's the full picture:

---

### 1. Scrape Manager ‚Äî The Coordinator

**`manager.go`** ‚Äî The `Manager` struct is the top-level orchestrator:

```
Manager
‚îú‚îÄ‚îÄ scrapeConfigs   map[string]*ScrapeConfig     // job_name ‚Üí config
‚îú‚îÄ‚îÄ scrapePools     map[string]*scrapePool        // job_name ‚Üí pool of loops
‚îú‚îÄ‚îÄ targetSets      map[string][]*targetgroup.Group
‚îú‚îÄ‚îÄ buffers         *pool.Pool                    // reusable byte slices
‚îî‚îÄ‚îÄ triggerReload   chan struct{}                  // signal reload
```

**Flow:**
1. `Run(tsets)` receives target groups from the discovery manager
2. Stores them in `targetSets`, then signals `triggerReload`
3. A background goroutine calls `reload()`, which syncs each pool concurrently
4. `ApplyConfig()` hot-reloads scrape configs and rebuilds affected pools

Each **scrape pool** (`scrape.go`) manages one loop per target:

```go
type scrapePool struct {
    config          *config.ScrapeConfig
    loops           map[uint64]loop          // one per unique target
    activeTargets   map[uint64]*Target
    symbolTable     *labels.SymbolTable      // interns repeated label strings
}
```

`Sync([]*targetgroup.Group)` hashes each target by its labels ‚Äî new targets get a loop started, removed targets get stopped.

### 2. Parsing the Text Format

**`textparse_interface.go`** defines the `Parser` interface that all format parsers implement:

```go
type Parser interface {
    Next() (Entry, error)              // advance to next entry
    Series() ([]byte, *int64, float64) // metric name bytes, optional timestamp, value
    Histogram() ([]byte, *int64, *histogram.Histogram, *histogram.FloatHistogram)
    Help() ([]byte, []byte)            // metric name, help text
    Type() ([]byte, model.MetricType)  // metric name, type (counter/gauge/etc.)
    Labels(l *labels.Labels)
    Exemplar(l *exemplar.Exemplar) bool
}
```

Entry types map to lines in the exposition format:
- `EntryType` ‚Üí `# TYPE http_requests_total counter`
- `EntryHelp` ‚Üí `# HELP http_requests_total Total requests`
- `EntrySeries` ‚Üí `http_requests_total{method="GET"} 1234`
- `EntryHistogram` ‚Üí native histogram samples

Three formats are supported: `text/plain`, `application/openmetrics-text`, and protobuf. The parser is selected by `extractMediaType()` on the HTTP response.

### 3. Counter & Gauge Internals

**`prom_counter.go`** ‚Äî Counters use a dual-value atomic trick for performance:

```go
type counter struct {
    valBits uint64           // atomic float64 bits (for Add with fractional values)
    valInt  uint64           // atomic uint64 (fast path for Inc)
    desc    *Desc
    exemplar atomic.Value    // *dto.Exemplar
}
```

- **`Inc()`** ‚Üí `atomic.AddUint64(&c.valInt, 1)` ‚Äî single instruction, no CAS loop
- **`Add(v)`** ‚Üí if integer-representable, adds to `valInt`; otherwise CAS-loops on `valBits`
- **`Write()`** ‚Üí final value = `float64(valInt) + Float64frombits(valBits)` ‚Äî merges both accumulators
- Panics if `v < 0` (counters only go up)

**`prom_gauge.go`** ‚Äî Gauges are simpler, just atomic float64:

```go
type gauge struct {
    valBits uint64           // atomic float64 bits
    desc    *Desc
}
```

- **`Set(val)`** ‚Üí `atomic.StoreUint64(&g.valBits, math.Float64bits(val))`
- **`Add(val)`** ‚Üí CAS loop: load old bits ‚Üí convert to float ‚Üí add ‚Üí store new bits
- `Inc()`, `Dec()`, `Sub()` all delegate to `Add()`

### 4. Registry & MetricFamily Management

**`prom_registry.go`** ‚Äî The `Registry` is the central collection point:

```go
type Registry struct {
    collectorsByID       map[uint64]Collector   // keyed by descriptor hash
    descIDs              map[uint64]struct{}     // all registered descriptor IDs
    dimHashesByName      map[string]uint64       // ensures label consistency per name
    uncheckedCollectors  []Collector             // collectors that skip Describe()
}
```

**Registration** (`Register()`):
- Calls `Describe()` to get all descriptors from the collector
- Validates: no duplicate names, consistent label sets, valid descriptor
- Stores by descriptor ID hash

**Gathering** (`Gather()`):
- Spawns goroutines to call `Collect()` on all registered collectors concurrently
- Each collector sends `Metric` values on a channel
- Metrics are grouped into `MetricFamily` structs (one per metric name)
- Validates type consistency and label uniqueness within each family
- Returns `[]*dto.MetricFamily` sorted lexicographically by name

### 5. End-to-End Data Flow

```
Discovery Manager
    ‚îÇ  target groups (endpoints to scrape)
    ‚ñº
Scrape Manager (manager.go)
    ‚îÇ  reload() ‚Üí sync pools concurrently
    ‚ñº
Scrape Pool (scrape.go)
    ‚îÇ  one loop per target, managed by label hash
    ‚ñº
Scrape Loop
    ‚îÇ  HTTP GET ‚Üí target:port/metrics
    ‚ñº
Text Parser (textparse_interface.go)
    ‚îÇ  Next() loop ‚Üí Series()/Histogram()/Help()/Type()
    ‚ñº
Relabeling (relabel.go)
    ‚îÇ  replace, keep, drop, hashmod, labelmap, etc.
    ‚ñº
Storage Appender (interface.go)
    ‚îÇ  Append(ref, labels, timestamp, value)
    ‚îÇ  AppendHistogram(ref, labels, timestamp, histogram)
    ‚îÇ  AppendExemplar(ref, labels, exemplar)
    ‚ñº
Commit() ‚Üí TSDB Head (in-memory + WAL)
```

Key architectural patterns throughout:
- **Object pooling** (`pool.Pool`) to reduce GC pressure on byte buffers
- **Symbol tables** (`labels.SymbolTable`) to intern repeated label strings across targets
- **Staleness markers** (`StaleNaN = 0x7ff0000000000002`) appended when a series disappears between scrapes
- **SeriesRef** ‚Äî ephemeral handles returned by `Append()` for fast subsequent appends to the same series, avoiding repeated label lookups
