Here's how histogram bucket counting works in this codebase. There are two layers: **classic (fixed-boundary) buckets** and **native (sparse/exponential) buckets**.

## Classic Bucket Counting

Classic histograms use pre-defined upper bounds. The relevant bucket creation functions are in `prom_histogram.go`:

```go
func LinearBuckets(start, width float64, count int) []float64        // line 295
func ExponentialBuckets(start, factor float64, count int) []float64   // line 315
func ExponentialBucketsRange(minBucket, maxBucket float64, count int) []float64  // line 339
```

- `LinearBuckets` ã creates `count` buckets where each boundary = previous + `width`
- `ExponentialBuckets` ã creates `count` buckets where each boundary = previous û `factor`
- `ExponentialBucketsRange` ã creates `count` buckets exponentially spaced between `min` and `max`

## Native (Sparse) Bucket Observation

The core counting logic is in `histogramCounts.observe` (`prom_histogram.go:655`):

```go
func (hc *histogramCounts) observe(v float64, bucket int, doSparse bool)
```

This does two things:
1. **Classic**: atomically increments `hc.buckets[bucket]` (pre-determined index)
2. **Native/sparse**: computes the bucket key using `math.Frexp(math.Abs(v))` to decompose the value into fraction and exponent, then:
   - For **positive schemas** (`schema > 0`): binary searches `nativeHistogramBounds[schema]` for sub-exponential bucket placement: `key = SearchFloat64s(bounds, frac) + (exp-1)*len(bounds)`
   - For **non-positive schemas** (`schema <= 0`): uses `key = (exp + offset) >> -schema` to merge exponential buckets
   - Values within `zeroThreshold` go to the **zero bucket** (`atomic.AddUint64(&hc.nativeHistogramZeroBucket, 1)`)
   - Positive/negative values go to separate `sync.Map` stores via `addToBucket()`

## Bucket Limiting

When too many native buckets are created, `limitBuckets` (`prom_histogram.go:919`) kicks in:

```go
func (h *histogram) limitBuckets(counts *histogramCounts, value float64, bucket int)
func (h *histogram) maybeWidenZeroBucket(hot, cold *histogramCounts) bool  // line 1017
func (h *histogram) doubleBucketWidth(hot, cold *histogramCounts)          // line 1087
```

These reduce resolution by either widening the zero bucket to absorb nearby buckets, or halving the schema (doubling bucket width).

## Validation

Bucket counts are validated via `validateCount` (`prom_histogram.go:1871`):

```go
func validateCount(sum float64, count uint64, negativeBuckets, positiveBuckets map[int]int64, zeroBucket uint64) error
```

And in the data model layer, `Histogram.Validate` (`histogram.go:426`) checks that `nCount + pCount + ZeroCount == Count` (or `>=` if Sum is NaN).

## Iteration
