Here's a comprehensive walkthrough of how Prometheus metrics scraping and collection works in this codebase.

---

## 1. Scrape Manager — Top-Level Orchestrator

The entry point is the `Manager` struct in `manager.go:133-156`:

```go
type Manager struct {
    scrapeConfigs map[string]*config.ScrapeConfig
    scrapePools   map[string]*scrapePool
    targetSets    map[string][]*targetgroup.Group
    triggerReload chan struct{}
    // ...
}
```

**Data flow:**

1. **`Manager.Run()`** (`manager.go:158-179`) listens on a channel of target group updates from service discovery. When new target sets arrive, it calls `updateTsets()` and signals `triggerReload`.

2. **`Manager.reloader()`** (`manager.go:186-209`) runs in a background goroutine with a ticker (default 5s). When it receives a reload signal, it calls `Manager.reload()`.

3. **`Manager.reload()`** (`manager.go:211-245`) creates a `scrapePool` for each scrape config that doesn't have one yet, then calls `sp.Sync(groups)` in parallel goroutines to reconcile active targets.

4. **`Manager.ApplyConfig()`** (`manager.go:278-344`) handles config reloads — it updates scrape configs, cleans up removed pools, and reloads existing pools concurrently (bounded by `GOMAXPROCS`).

## 2. Scrape Pool — Per-Job Target Management

`scrapePool` (`scrape.go:83-116`) manages all targets for a single scrape job:

```go
type scrapePool struct {
    config        *config.ScrapeConfig
    client        *http.Client
    loops         map[uint64]loop           // hash → scrape loop
    activeTargets map[uint64]*Target        // hash → target
    appendable    storage.Appendable        // where samples go
    // ...
}
```

**`scrapePool.sync()`** (`scrape.go:494-525`) is the key reconciliation method. For each discovered target, it either reuses an existing loop or creates a new one via `sp.newLoop()`. Old loops not in the new target set are stopped. New loops are started with `go l.run(nil)`.

## 3. Scrape Loop — The Per-Target Ticker

`scrapeLoop` (`scrape.go:822-881`) is the core scraping engine for a single target:

```go
type scrapeLoop struct {
    scraper   scraper        // does the HTTP GET
    cache     *scrapeCache   // tracks series across scrapes
    interval  time.Duration
    timeout   time.Duration
    appendable storage.Appendable
    // ...
}
```

**`scrapeLoop.run()`** (`scrape.go:1234-1299`) is the main loop:
1. Waits for an initial offset (to spread scrape load across HA setups)
2. Runs a `time.Ticker` at the configured interval
3. Each tick calls `sl.scrapeAndReport(last, scrapeTime, errc)` which performs the HTTP scrape and appends results

The actual HTTP request is done by `targetScraper.scrape()` (`scrape.go:735-752`) — a simple `GET` with `Accept`, `Accept-Encoding`, `User-Agent`, and timeout headers.

## 4. Parsing the Text Format

After the HTTP response body is read, `scrapeLoopAppender.append()` (`scrape.go:1562-1624`) creates a parser:

```go
p, err := textparse.New(b, contentType, sl.symbolTable, textparse.ParserOptions{...})
```

The `Parser` interface (`textparse_interface.go:28-87`) is a streaming parser that yields entries one at a time:

```go
type Parser interface {
    Series() ([]byte, *int64, float64)
    Histogram() ([]byte, *int64, *histogram.Histogram, *histogram.FloatHistogram)
    Help() ([]byte, []byte)
    Type() ([]byte, model.MetricType)
    Labels(l *labels.Labels)
    Exemplar(l *exemplar.Exemplar) bool
    Next() (Entry, error)
}
```

Entry types: `EntrySeries`, `EntryHistogram`, `EntryType`, `EntryHelp`, `EntryUnit`, `EntryComment`.

The append loop (`scrape.go:1625-1762`) iterates the parser:
- **`EntryType`/`EntryHelp`/`EntryUnit`** → cached as metadata in `sl.cache`
- **`EntrySeries`** → calls `p.Series()` to get metric name bytes, optional timestamp, and float64 value
- **`EntryHistogram`** → calls `p.Histogram()` for native histogram data

For each sample, labels are resolved (from cache or freshly parsed), mutated through relabeling, and then appended to storage via `app.Append(ref, lset, t, val)` or `app.AppendHistogram(...)`.

## 5. Counters and Gauges — Internal Tracking

### Counter (`prom_counter.go:103-120`)

```go
type counter struct {
    valBits uint64  // float64 bits for fractional adds
    valInt  uint64  // exact integer accumulator
    desc    *Desc
    labelPairs []*dto.LabelPair
    exemplar   atomic.Value
}
```

- **`Inc()`** (`prom_counter.go:151`): `atomic.AddUint64(&c.valInt, 1)` — fast path for integer increments
- **`Add(v)`** (`prom_counter.go:126-144`): If `v` is an exact integer, uses `atomic.AddUint64` on `valInt`. Otherwise, CAS loop on `valBits` for float64 addition. Panics if `v < 0`.
- **`get()`** (`prom_counter.go:155-159`): Returns `float64(valInt) + Float64frombits(valBits)` — combines both accumulators

The dual-accumulator design avoids floating-point precision loss for the common case of integer increments.

### Gauge (`prom_gauge.go:90-100`)

```go
type gauge struct {
    valBits uint64  // float64 bits
    desc    *Desc
    labelPairs []*dto.LabelPair
}
```

- **`Add(val)`** (`prom_gauge.go:122-130`): CAS loop — `atomic.CompareAndSwapUint64` with `Float64bits/Float64frombits`
- **`Inc()`/`Dec()`**: Delegate to `Add(1)`/`Add(-1)`
- **`Set(v)`**: Direct `atomic.StoreUint64`

Both use lock-free atomics for thread safety.

## 6. Registry — Managing Metric Families

### Registry struct (`prom_registry.go:253-267`)

```go
type Registry struct {
    mtx                 sync.RWMutex
    collectorsByID      map[uint64]Collector  // XOR of desc IDs → collector
    descIDs             map[uint64]struct{}
    dimHashesByName     map[string]uint64
    uncheckedCollectors []Collector
}
```

### Registration (`prom_registry.go:269-363`)

`Register(c Collector)`:
1. Calls `c.Describe(descChan)` to get all `Desc` descriptors
2. Validates each descriptor (unique fqName + constLabels combo)
3. Checks label name/help string consistency against previously registered descriptors
4. Computes `collectorID` as XOR of all desc IDs
5. Stores in `collectorsByID[collectorID] = c`
6. Collectors with no descriptors go into `uncheckedCollectors`

### Gathering (`prom_registry.go:411-560`)

`Registry.Gather()` is the collection pipeline:

1. Launches **concurrent collector workers** — one goroutine per collector, each calling `collector.Collect(metricChan)`
2. Drains metrics from two channels (checked and unchecked) in a select loop
3. Each metric goes through **`processMetric()`** (`prom_registry.go:619-725`):
   - Calls `metric.Desc()` to get the descriptor
   - Calls `metric.Write(dtoMetric)` to serialize into protobuf
   - Groups into `metricFamiliesByName` map, auto-detecting type (COUNTER/GAUGE/SUMMARY/HISTOGRAM/UNTYPED)
   - Validates consistency — same name must have same help string and type
   - Deduplicates via `metricHashes`
4. Returns `NormalizeMetricFamilies(metricFamiliesByName)` — sorted, validated MetricFamily slice

The `Gatherers` type (`prom_registry.go:727-803`) composes multiple `Gatherer` instances, merging results and deduplicating across registries.

## Summary Data Flow

```
Discovery → Manager.Run() → triggerReload → Manager.reload()
                                                  │
                                    ┌──────────────┘
                                    ▼
                             scrapePool.Sync()
                              │  creates/stops loops
                              ▼
                         scrapeLoop.run()
                          │  ticker every interval
                          ▼
                     targetScraper.scrape()  ← HTTP GET /metrics
                          │
                          ▼
                     textparse.Parser.Next()  ← streaming parse
                          │
                          ▼
                  scrapeLoopAppender.append()
                   │  label mutation, relabeling
                   ▼
              storage.Appender.Append(ref, labels, timestamp, value)
                          │
                          ▼
                   TSDB Head (in-memory)
```

On the client library side (for exposing metrics):
```
Counter/Gauge.Add() → atomic update → Collect(ch) → Write(dto.Metric)
     → Registry.Gather() → processMetric() → MetricFamily protobuf → HTTP exposition
```
