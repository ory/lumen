Here's how TSDB compaction works end-to-end:

---

## The `Compactor` Interface (`compact.go:54-77`)

```go
type Compactor interface {
    Plan(dir string) ([]string, error)
    Write(dest string, b BlockReader, mint, maxt int64, base *BlockMeta) ([]ulid.ULID, error)
    Compact(dest string, dirs []string, open []*Block) ([]ulid.ULID, error)
}
```

Three responsibilities:
- **`Plan`** — scans block directories and returns which ones should be compacted together
- **`Write`** — persists a single `BlockReader` (typically the in-memory Head) to a new on-disk block
- **`Compact`** — merges multiple existing on-disk blocks into one new block

---

## `LeveledCompactor` (`compact.go:80-93`)

The concrete implementation. Key fields:

| Field | Purpose |
|---|---|
| `ranges []int64` | Exponential time ranges (e.g. 2h, 6h, 18h) that define compaction levels |
| `mergeFunc` | Strategy for merging overlapping chunk series (vertical compaction) |
| `enableOverlappingCompaction` | Whether to merge time-overlapping blocks |
| `blockExcludeFunc` | Optional filter to skip blocks (used by Mimir/Thanos) |

### Planning algorithm (`plan`, line 279)

1. Sort blocks by `MinTime`
2. **First priority**: find overlapping blocks via `selectOverlappingDirs` — if any block's `MinTime < previous block's MaxTime`, group them for vertical compaction
3. **Second priority**: level-based compaction via `selectDirs` — uses `splitByRange` to group blocks into aligned time windows at each range level, picks the first group of 2+ blocks that spans a full range or is behind the newest block
4. **Third priority**: tombstone cleanup — any block with >5% tombstones or fully deleted series

### Compact flow (`Compact`, line 485)

1. Read each block's `meta.json`, reuse already-open `*Block` objects when possible
2. Call `CompactBlockMetas` — merges metadata, increments compaction level, records parent blocks
3. Call `write` — creates a temp dir, opens chunk/index writers, calls `BlockPopulator.PopulateBlock` to iterate all series across input blocks, merge them, and write chunks + index entries
4. Atomically rename temp dir to final block dir
5. If the result has 0 samples, mark source blocks as `Deletable`
6. On failure, mark source blocks' compaction as `Failed`

### `PopulateBlock` (`DefaultBlockPopulator`, line 793)

The core merge logic: opens index/chunk/tombstone readers for each input block, creates `ChunkSeriesSet` iterators, merges symbols, then iterates all series writing chunks and index entries while accumulating stats into the new block's `BlockMeta`.

## How the DB Triggers Compaction

### The `run` loop (`db.go:1175`)

The DB starts a background goroutine that runs a select loop on two channels:

1. **`BlockReloadInterval` timer** (default 1 min) — reloads blocks from disk, then sends on `compactc` to trigger compaction. Also checks stale series ratio for early compaction.
2. **`compactc` channel** — when received, calls `db.Compact(ctx)` if `autoCompact` is true. Uses exponential backoff on failure.

External triggers (e.g. `dbAppender.Commit`) also send on `compactc` to signal the head is ready.

### `DB.Compact` (`db.go:1414`) — the main entry point

Three phases executed sequentially:

1. **Head compaction** — loops while `head.compactable()` returns true:
   - Builds a `RangeHead` from `[mint, maxt)` using the head's chunk range
   - Waits for overlapping appenders to finish
   - Calls `compactHead` → `compactor.Write(db.dir, head, ...)` — writes head data to a new level-1 block
   - Reloads blocks, truncates head memory and WAL

2. **OOO (out-of-order) head compaction** — if the head was compacted, also compact out-of-order data via `compactOOOHead`

3. **Block compaction** via `compactBlocks` (`db.go:1721`):
   - Loops calling `compactor.Plan(db.dir)` until no more plans
   - For each plan, calls `compactor.Compact(db.dir, plan, db.blocks)`
   - Reloads blocks after each compaction
   - Aborts early if the head becomes compactable again (head has priority)

### Control mechanisms

- `DisableCompactions` / `EnableCompactions` (`db.go:2243-2256`) — toggle `autoCompact` flag
- `compactCancel` context — allows canceling a running compaction on shutdown
- `cmtx` mutex — ensures compactions and deletions don't run simultaneously
- Optional `CompactionDelay` — delays compaction start to allow backups of fresh data

---

**In summary**: the DB's background loop periodically triggers `Compact`, which first flushes the in-memory Head to level-1 blocks via `Write`, then repeatedly calls `Plan` + `Compact` to merge small blocks into larger ones following the exponential range levels — all orchestrated through the `Compactor` interface with `LeveledCompactor` as the default implementation.
