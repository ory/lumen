## How Histogram Bucket Counting Works

The core counting mechanism lives in `Histogram.Validate()` (`histogram.go:426`). It works by accumulating bucket counts from positive, negative, and zero buckets, then verifying they match `h.Count`.

### The counting flow:

1. **Delta-encoded buckets** — `PositiveBuckets` and `NegativeBuckets` are delta-encoded `[]int64` slices. The first element is an absolute count; each subsequent element is a delta relative to the previous one. See `histogram.go:70-73`.

2. **Accumulation** — The `regularBucketIterator.Next()` (`histogram.go:496-528`) decodes deltas into running counts:
   ```go
   r.currCount += r.buckets[r.bucketsIdx]  // line 524
   ```
   Each call to `Next()` adds the next delta to `currCount`, reconstructing the absolute count for that bucket.

3. **Cumulative counting** — The `cumulativeBucketIterator.Next()` (`histogram.go:550-607`) builds a running cumulative total:
   ```go
   c.currCount += c.h.PositiveBuckets[c.posBucketsIdx]   // line 590 — decode delta
   c.currCumulativeCount += uint64(c.currCount)            // line 591 — accumulate
   ```

4. **Validation** — `Validate()` (`histogram.go:426-479`) calls `checkHistogramBuckets` for both positive and negative sides to sum all bucket observations into `nCount` and `pCount`, then checks:
   ```go
   sumOfBuckets := nCount + pCount + h.ZeroCount  // line 467
   ```
   - If `Sum` is NaN: `sumOfBuckets <= h.Count` (lower bound, since NaN observations increment Count but not buckets)
   - Otherwise: `sumOfBuckets == h.Count` (exact match required)

### Spans control which buckets exist

The `Span` struct (`histogram.go:83-89`) maps the sparse bucket layout. Each span has an `Offset` (gap from previous span or starting index) and `Length` (number of consecutive buckets). The iterators walk spans to determine which bucket indices are populated, skipping gaps between spans.
