The core of histogram bucket counting lives in `prom_histogram.go:652`.

### How it works

**`histogramCounts.observe(v float64, bucket int, doSparse bool)`** — `prom_histogram.go:652`

This method handles all bucket counting for a single observation:

1. **Fixed-width (classic) buckets** — if the pre-computed `bucket` index falls within the bucket array, it atomically increments that slot:
   ```go
   if bucket < len(hc.buckets) {
       atomic.AddUint64(&hc.buckets[bucket], 1)
   }
   ```

2. **Sparse (native) buckets** — when `doSparse` is true, the value is placed into a logarithmic bucket using a schema-dependent key:
   - For `schema > 0`: binary searches the `nativeHistogramBounds` table using `math.Frexp` to get the fractional exponent, then maps to a key.
   - For `schema <= 0`: uses the raw exponent, offset-shifted by the schema.
   - Infinities are handled by clamping to `±MaxFloat64` and incrementing the key by 1.
   - The key routes to positive buckets, negative buckets, or the zero bucket:
     ```go
     case v > zeroThreshold:  addToBucket(&hc.nativeHistogramBucketsPositive, key, 1)
     case v < -zeroThreshold: addToBucket(&hc.nativeHistogramBucketsNegative, key, 1)
     default:                 atomic.AddUint64(&hc.nativeHistogramZeroBucket, 1)
     ```

3. **Count is incremented last** (atomically) as the signal that the observation is complete — this is an intentional ordering guarantee.

**Validation** — `histogram.go:470` — after the fact, `Validate` sums all bucket counts and checks they match `h.Count`, returning `ErrHistogramCountMismatch` or `ErrHistogramCountNotBigEnough` if not.
