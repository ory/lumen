Here's a comprehensive explanation of how PromQL query evaluation works across the codebase:

---

## PromQL Query Evaluation Pipeline

### 1. Entry Points: `Engine` and `Query`

The `Engine` exposes two query constructors:

```go
// engine.go:550
func (ng *Engine) NewRangeQuery(ctx context.Context, q storage.Queryable, opts QueryOpts,
    qs string, start, end time.Time, interval time.Duration) (Query, error)

func (ng *Engine) NewInstantQuery(ctx context.Context, q storage.Queryable, opts QueryOpts,
    qs string, ts time.Time) (Query, error)
```

Both parse the query string into an AST via `ng.parser.ParseExpr(qs)`, run `PreprocessExpr` to fold step-invariant subexpressions, and return a `*query`. The caller then calls `query.Exec(ctx)` to trigger evaluation.

---

### 2. AST Node Hierarchy

All expression nodes implement the `Expr` interface (`ast.go:76`):

```go
type Expr interface {
    Node
    Type() ValueType   // ValueTypeVector, ValueTypeMatrix, ValueTypeScalar, ValueTypeString
    PromQLExpr()       // marker method to prevent accidental implementation
}
```

Concrete node types:

| Type | Description |
|------|-------------|
| `AggregateExpr` | `sum`, `avg`, `count`, etc. with `Op`, `Expr`, `Grouping`, `Without` |
| `BinaryExpr` | arithmetic/comparison with `LHS`, `RHS`, `VectorMatching` |
| `Call` | function call with `Func *Function`, `Args Expressions` |
| `VectorSelector` | instant series lookup, holds `LabelMatchers` and pre-expanded `Series []storage.Series` |
| `MatrixSelector` | range vector, wraps a `VectorSelector` with a `Range` |
| `SubqueryExpr` | nested range over an instant vector |
| `StepInvariantExpr` | wraps expressions evaluated once then duplicated across steps |
| `NumberLiteral`, `StringLiteral`, `UnaryExpr`, `ParenExpr` | leaves and wrappers |

---

### 3. Execution Flow: `exec` → `execEvalStmt` → `evaluator.eval`

```
query.Exec(ctx)
  └─ Engine.exec(ctx, q)                          // engine.go:669
       └─ Engine.execEvalStmt(ctx, query, stmt)   // engine.go:772
            ├─ ng.populateSeries(...)             // pre-fetch storage series
            └─ evaluator.Eval(ctx, expr)          // dispatch to eval()
```

**`execEvalStmt`** (`engine.go:772`):
1. Computes `mint`/`maxt` across all selectors via `FindMinMaxTime`
2. Opens a `storage.Querier` for that time range
3. Calls `ng.populateSeries` — walks the AST, calls `querier.Select(...)` on every `VectorSelector`, storing `SeriesSet` into `n.UnexpandedSeriesSet`
4. Creates an `evaluator` struct with timestamps and interval
5. Calls `evaluator.Eval(ctx, expr)` — returns `(parser.Value, annotations.Annotations, error)`

For instant queries: `start == end`, `interval = 1`, single evaluation pass.
For range queries: `start < end`, stepped over `interval`.

---

### 4. The `evaluator.eval` Dispatch Switch

The core recursive method (`engine.go:1904`):

```go
func (ev *evaluator) eval(ctx context.Context, expr parser.Expr) (parser.Value, annotations.Annotations)
```

It type-switches on the AST node and dispatches accordingly:

```go
switch e := expr.(type) {
case *parser.AggregateExpr:
    // Sort grouping labels, call ev.rangeEval with aggregation funcCall
    val, ws := ev.eval(ctx, e.Expr)           // recurse on inner expr
    result, ws := ev.rangeEvalAgg(...)

case *parser.Call:
    // Range vector function path (matrix arg) vs instant vector path
    // Calls ev.rangeEval(ctx, nil, wrappedFuncCall, e.Args...)

case *parser.BinaryExpr:
    // scalar×scalar → scalarBinop
    // vector×vector → ev.rangeEval with VectorAnd/Or/Unless/arithmeticMatching
    // scalar×vector or vector×scalar → appropriate rangeEval wrapper

case *parser.VectorSelector:
    // Expand series, sample at each step with lookback delta

case *parser.MatrixSelector:
    // Only at instant eval; calls ev.matrixSelector()

case *parser.SubqueryExpr:
    // Create child evaluator with adjusted start/interval, recurse

case *parser.StepInvariantExpr:
    // Evaluate once at startTimestamp, duplicate results across all steps

case *parser.UnaryExpr:
    // Negate floats/histograms element-wise after recursing

case *parser.ParenExpr:
    return ev.eval(ctx, e.Expr)    // transparent passthrough

case *parser.NumberLiteral, *parser.StringLiteral:
    // Return scalar/string directly
}
```

### 5. Function Registration and Dispatch

**Type** (`functions.go:39`):

```go
type FunctionCall func(
    vectorVals []Vector,    // evaluated instant vector args
    matrixVals Matrix,      // evaluated range vector arg (if any)
    args parser.Expressions,// original AST args (for string literals, etc.)
    enh *EvalNodeHelper,    // helper with pre-allocated output vector
) (Vector, annotations.Annotations)
```

**Registry** (`functions.go:2152`) — a flat `map[string]FunctionCall`:

```go
var FunctionCalls = map[string]FunctionCall{
    "abs":                funcAbs,
    "rate":               funcRate,
    "histogram_quantile": funcHistogramQuantile,
    "increase":           funcIncrease,
    // ~80+ entries...
}
```

**Invocation** — the `Call` branch in `eval` calls `ev.rangeEval(...)`, which for each timestep:
1. Slices the pre-evaluated matrices down to vectors for that step
2. Constructs `enh.Out` (reused output buffer)
3. Calls the `FunctionCall` with `(vectorVals, matrixVals, args, enh)`
4. Accumulates results into the output `Matrix`

Range vector functions (e.g. `rate`, `delta`) receive the raw `Matrix` with one series at a time, iterated by the engine per series per step.

---

### 6. `rangeEval` — The Step-Loop Engine

`evaluator.rangeEval` (`engine.go:1384`) is the universal step-looping primitive:

```go
func (ev *evaluator) rangeEval(
    ctx context.Context,
    matching *parser.VectorMatching,
    funcCall func([]Vector, Matrix, [][]EvalSeriesHelper, *EvalNodeHelper) (Vector, annotations.Annotations),
    exprs ...parser.Expr,
) (Matrix, annotations.Annotations)
```

It:
1. Evaluates all `exprs` into matrices upfront (one call to `ev.eval` each)
2. Loops `ts` from `startTimestamp` to `endTimestamp` by `interval`
3. At each step, extracts the current `Vector` slice from each matrix
4. Calls `funcCall` with those vectors → gets back a `Vector`
5. Merges output vectors into a `seriess` map keyed by label hash → final `Matrix`

---

### 7. Alert and Recording Rules

Both rule types implement the `Rule` interface and share the `QueryFunc` abstraction:

```go
type QueryFunc func(ctx context.Context, q string, t time.Time) (promql.Vector, error)
```

**`AlertingRule.Eval`** (`alerting.go:380`):

```go
func (r *AlertingRule) Eval(ctx context.Context, queryOffset time.Duration,
    ts time.Time, query QueryFunc, externalURL *url.URL, limit int) (promql.Vector, error)
```

1. Calls `query(ctx, r.vector.String(), ts.Add(-queryOffset))` — fires a PromQL instant query
2. Each sample in the result vector becomes an `Alert` candidate
3. Expands Go templates for labels/annotations using the sample value
4. Updates the `r.active` map: new fingerprints → `StatePending`; after `holdDuration` → `StateFiring`; absent fingerprints → resolved

**`RecordingRule.Eval`** (`recording.go:84`):

```go
func (rule *RecordingRule) Eval(ctx context.Context, queryOffset time.Duration,
    ts time.Time, query QueryFunc, _ *url.URL, limit int) (promql.Vector, error)
```

1. Same `query(...)` call → gets a `Vector`
2. Rewrites each sample's metric name to `rule.name` and merges `rule.labels`
3. Checks for duplicate labelsets
4. Returns the vector to be written back to storage

Both rule types are driven by a `Group.run()` loop that calls `.Eval()` on a ticker matching the group's `interval`. The `QueryFunc` passed in is typically `rules.EngineQueryFunc`, which calls `Engine.NewInstantQuery(...).Exec(ctx)` — feeding results back through the full engine pipeline described above.

### End-to-End Summary

```
HTTP /query_range
  → API.queryRange
      → Engine.NewRangeQuery (parse, PreprocessExpr)
          → query.Exec
              → Engine.execEvalStmt
                  → populateSeries (storage.Select per VectorSelector)
                  → evaluator.eval (recursive AST dispatch)
                      → rangeEval (step loop)
                          → FunctionCall (per-step function)
                      → AggregateExpr / BinaryExpr / ...
                  → Matrix result

Rule evaluation (ticker loop)
  → Group.run → Rule.Eval (AlertingRule / RecordingRule)
      → QueryFunc → Engine.NewInstantQuery → query.Exec → same pipeline above
      → AlertingRule: update active alert state machine
      → RecordingRule: rewrite metric name, return samples for remote write
```
