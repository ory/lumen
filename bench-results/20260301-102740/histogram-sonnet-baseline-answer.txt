Here's how histogram bucket counting works in this codebase:

## Histogram Bucket Counting

The implementation is in `testdata/fixtures/go/prom_histogram.go` and follows a two-phase flow:

### 1. Finding the bucket — `findBucket`
```go
// prom_histogram.go ~line 866
func (h *histogram) findBucket(v float64) int
```
Uses **linear search** for small arrays (n < 35) or **binary search** (`sort.SearchFloat64s`) for larger ones against `h.upperBounds`.

### 2. Public entry point — `Observe`
```go
// prom_histogram.go ~line 766
func (h *histogram) Observe(v float64) {
    h.observe(v, h.findBucket(v))
}
```

### 3. Hot/cold count dispatch — `histogram.observe`
```go
// prom_histogram.go ~line 899
func (h *histogram) observe(v float64, bucket int)
```
Uses a hot/cold double-buffer scheme via `countAndHotIdx` — the top bit selects which `histogramCounts` is "hot".

### 4. Actual counting — `histogramCounts.observe`
```go
// prom_histogram.go ~line 652
func (hc *histogramCounts) observe(v float64, bucket int, doSparse bool)
```
Two counting paths:
- **Classic buckets**: `atomic.AddUint64(&hc.buckets[bucket], 1)` — simple atomic array increment
- **Native/sparse buckets**: delegates to `addToBucket` using `sync.Map` + atomics for positive/negative/zero buckets

### 5. Sparse bucket helper — `addToBucket`
```go
// prom_histogram.go ~line 1532
func addToBucket(buckets *sync.Map, key int, increment int64) bool
```
Returns `true` if a new bucket was created (so the total bucket count can be incremented separately).

### 6. Exposition — cumulative counts
At scrape time (`Write`), classic bucket counts are accumulated cumulatively: each bucket's count adds to all prior counts, producing the standard Prometheus cumulative histogram format.
